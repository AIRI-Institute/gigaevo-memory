# Chain content conventions

GigaEvo Memory stores every chain's `content` as opaque JSON â€” the
backend imposes no schema beyond "is a JSON object". This document
defines the **CARE convention** for the `content["metadata"]` block, so
multiple clients (CARE TUI, MAGE generator, evolution platform,
gigaevo-core) can co-operate without bespoke per-client logic.

A chain that follows the convention can be:

- Re-run by CARE with the same inputs (`task_description` +
  `context_files`).
- Rendered in the CARE library with a friendly name, description,
  tags, and provenance ("generated by MAGE", "generated by user").
- Audited against its upstream MAGE generation via `mage_metadata`.

## Schema

The whole CARE block lives under `content["metadata"]`. Other clients
may add their own keys alongside it â€” the CARE keys are listed below.

```json

  "max_workers": 2,
  "search_config": { "...": "..." },
  "steps": [ "...": "..." ],
  "metadata": {

    "context_files": [
```

### Field reference

| Key                | Type                       | Required | Purpose                                                                  |
|--------------------|----------------------------|----------|--------------------------------------------------------------------------|
| `task_description` | `str \| None`              |          | Verbatim user query that prompted the chain. Re-fed into `outer_context`. |
| `context_files`    | `list[ContextFileRef]`     |          | Files attached at generation time. Empty list when no files were picked. |
| `generated_by`     | `str \| None`              |          | `"mage"` / `"user"` / etc. Provenance tag.                               |
| `mage_metadata`    | `dict[str, Any] \| None`   |          | Full `MAGEMetadata.model_dump()` from the upstream generation.            |
| `display_name`     | `str \| None` (â‰¤ 200 char) |          | Human-friendly label for the library. Same role as `Entity.display_name`. |
| `description`      | `str \| None`              |          | Free-form description for the library tooltip / detail pane.             |
| `tags`             | `list[str]`                |          | User-facing tags. The reserved value `"favourite"` is shown as a star.   |

Every field is optional â€” `CareChainMetadata()` with no kwargs is a
valid (but useless) instance. Clients should treat missing fields as
"not specified", not as "explicitly cleared".

### `ContextFileRef`

| Key          | Type           | Required | Notes                                                       |

| `path`       | `str`          | âś“        | Absolute or workspace-relative path at generation time.     |

## Why `Entity.display_name` AND `metadata.display_name`?

Each appears in different contexts:

- `entities.display_name` (the DB column added by migration 003) is
  the **mutable, indexed** field the library lists query. Users edit
  it via `PATCH /v1/agents/{id}`.
- `metadata.display_name` (this convention) is the **content-embedded**
  value. It travels with the chain when the user exports a JSON dump
  or evolves the chain on the platform â€” keeping the chain JSON
  self-describing without a join against `entities`.

On save, CARE writes both. On read, CARE prefers `Entity.display_name`
(it's authoritative because PATCH never rewrites `content`).

## Helper API

Both the server-side request models and the client SDK expose
`CareChainMetadata` with two helpers:

```python
from gigaevo_memory import CareChainMetadata, ContextFileRef

# Build the metadata block from CARE state.
meta = CareChainMetadata(
    task_description="Develop a helper for a financier",
    context_files=[
        ContextFileRef(
            path="report.pdf",
            sha256="0a7f3b1c..." * 1,  # 64 hex chars
            size_bytes=152034,
            mime_type="application/pdf",
        ),
    ],
    generated_by="mage",
    display_name="Financier helper",
    tags=["finance", "favourite"],
)

# Embed into a fresh chain content dict.
chain_content = {
    "version": "1.1",
    "steps": [],  # populated by MAGE
}
chain_content = meta.merge_into_content(chain_content)

# Round-trip from a saved chain.
loaded = CareChainMetadata.from_chain_content(chain_content)
assert loaded == meta
```

The `merge_into_content()` helper preserves any non-CARE keys already
present in `content["metadata"]` (other clients can co-exist).

## Re-run flow

When the user picks an agent from the CARE library and chooses "Run
with same inputs":

1. CARE fetches the chain (`client.get_chain(entity_id)`).
2. Parses `CareChainMetadata.from_chain_content(chain.content)`.
3. For each `ContextFileRef` â€” checks whether the file still exists at
   `path` and the SHA matches. If yes, reads it into the
   `outer_context`. If no, prompts the user to re-pick (or skip).
4. Re-instantiates a `ReasoningContext` with the original
   `task_description` and the re-loaded file contents.
5. Records the run via `client.record_run(entity_id)` so the library's
   `last_run_at` / `run_count` updates.

## Compatibility notes

- The convention is **read-leniently / write-strictly**: clients
  should tolerate missing fields and unrecognised keys (forward
  compat), but emit only the documented shape (backward compat).
- The whole block is **optional**. Older chains that pre-date this
  convention will parse to an empty `CareChainMetadata`, which is
  fine â€” the library falls back to `Entity.display_name` etc.
- The convention version is **implicit** â€” additions are
  forward-compatible (just new optional fields). Breaking changes
  would require a `metadata_schema_version` key, but none are
  anticipated.
